域名解析系统DNS: Domain Name System

当用户在浏览器地址栏输入URL之后，首先在浏览器查询缓存，查询不到则在host文件和本地DNS查询。如果没有找到对应的IP地址则到路由器缓存中查询，以上客户端DNS缓存中均未发现，则进入ISP DNS缓存中查询，最后向远程DNS服务器发出DNS查询报文。（通过抓包可以查看DNS的数据包结构或通过dig @dns server + url 可以验证DNS的缓存机制）

DNS查询报文以UDP形式发出，服务器的默认端口是53，通过查看数据包DNS是应用层协议。

另外DNS也支持TCP访问，为了减轻单台DNS服务器的负载，一台服务器中的内容会传输到其他服务器中进行备份或分布式存储。这是DNS的区域传输就使用TCP进行。

DNS 记录类型：

A：直接记录域名对应的 IP 地址

CNAME：将域名解析到别的域名上

PTR： 反向 DNS，根据 IP 地址指向域名

MX：用于将邮件地址@后的域名指向邮件服务器

建立TCP连接：Transmission Control Protocol

三次握手：

​ （1）第一次握手：客户端发送数据包，服务器接收。

此时，可以得知客户端的发送能力和自己的接收能力均正常

​ （2）第二次握手：服务器端发送数据包，客户端接收数据包。

此时，客户端得知服务器端接收能力、发送能力正常，客户端接收能力发送能力正常。

​ （3）第三次握手：客户端发包、服务器接收

此时，服务器端得知客户端的发送、接收能力均正常。

在三次握手的过程中，第一次和第二次握手不可以携带数据，因为黑客可能会在第一次握手的过程中发送大量内容让服务器接收，导致服务器资源占用影响性能，第三次握手客户端已经知道服务器和自己接收、发送能力均正常因此可以携带数据。

四次挥手：

（1）第一次挥手

因此当主动方发送断开连接的请求（即FIN报文）给被动方时，仅仅代表主动方不会再发送数据报文了，但主动方仍可以接收数据报文。

（2）第二次挥手

被动方此时有可能还有相应的数据报文需要发送，因此需要先发送ACK报文，告知主动方“我知道你想断开连接的请求了”。这样主动方便不会因为没有收到应答而继续发送断开连接的请求（即FIN报文）。

（3）第三次挥手

被动方在处理完数据报文后，便发送给主动方FIN报文；这样可以保证数据通信正常可靠地完成。发送完FIN报文后，被动方进入LAST_ACK阶段（超时等待）。

（4）第四挥手

如果主动方及时发送ACK报文进行连接中断的确认，这时被动方就直接释放连接，进入可用状态

上图每个状态的含义：

LISTEN：侦听来自远方 TCP 端口的连接请求。

SYN-SENT：在发送连接请求后等待匹配的连接请求。

SYN-RECEIVED：在收到和发送一个连接请求后等待对连接请求的确认。

ESTABLISHED：代表一个打开的连接，数据可以传送给用户。

FIN-WAIT-1：等待远程 TCP 的连接中断请求，或先前的连接中断请求的确认。

FIN-WAIT-2：从远程 TCP 等待连接中断请求。

CLOSE-WAIT：等待从本地用户发来的连接中断请求。

CLOSING：等待远程 TCP 对连接中断的确认。

LAST-ACK：等待原来发向远程 TCP 的连接中断请求的确认。

TIME-WAIT：等待足够的时间以确保远程 TCP 接收到连接中断请求的确认。

CLOSED：没有任何连接状态。

Delay ACK 和 Nagle都是为了减少网络中tcp报文的数量。

流量控制和拥塞控制都是为了控制发送数据量的大小。

TCP可靠传输是如何保证的：超时重传、确认机制

TCP 与 UDP都是传输层协议

HTTP/HTTPS: Hyper Text Transfer Protocol

应用层协议

HTTP是无连接无状态的、一般构建于TCP/IP协议之上。可以分为请求和响应，默认端口号80。

无连接是限制每个连接只有一个请求的意思，在服务器处理完客户的请求，并收到客户的反应，即断开。通过这种方式可以节省传输时间。

HTTP缺点：

  • 请求信息明文传输，容易被窃听截取。
  • 数据的完整性未校验，容易被篡改
  • 没有验证对方身份，存在冒充危险

HTTPS:

HTTPS缺点：

  • HTTPS协议多次握手，导致页面的加载时间延长近50%；
  • HTTPS连接缓存不如HTTP高效，会增加数据开销和功耗；
  • 申请SSL证书需要钱，功能越强大的证书费用越高。
  • SSL涉及到的安全算法会消耗 CPU 资源，对服务器资源消耗较大。

底层网络协议：

ARP ( 地址解析协议 )：Adress Resolution Protocol

用于实现从 IP 地址到 MAC 地址的映射，即询问目标 IP 对应的 MAC 地址。基于分层、包封装考虑 ARP 属于网络层协议。

ARP 只能帮助主机寻找同网段的 IP 地址，当需要将数据包转发到不同网段的主机，数据包先将其转发给网关此时ARP 协议获得网关的 MAC 地址。

在获取 MAC 地址的工程中，主机先在网络内广播询问某 IP 地址的主机 MAC 地址是多少，非该 IP 地址的主机不做应答，对应 IP 地址的主机将其 MAC 地址单播回询问方并存储到ARP缓冲区的 ARP 列表

ARP 攻击：

由于主机在进行广播之后接收对应 IP 地址的 MAC 地址时，采用的是后到优先原则。所以在局域网中的攻击者可以发送 对应的 IP 地址以及自己的 MAC 地址，从而截获网络中的数据包。

ICMP（互联网控制消息协议）：Internet Control Message Protocol

用于TCP/IP 网络中发送控制信息，返回错误信息或者分析路由。

因为是帮助 IP 协议发送控制信息， IP 属于网络层，ICMP 同样属于网络层协议。

DHCP （动态主机设置协议）

局域网网络协议、使用 UDP 协议工作。主要用于内部网络或网络服务供应商自动分配 IP 地址给用户。

新主机进入网络后，发送广播报文申请 IP 地址信息的租约，并将客户端 MAC 地址和机器名记录在报文中。网络中所有的DHCP 服务器对广播做出响应，从地址池中提取未分配的 IP 地址并将其与 租约有效期、服务器 IP 地址一同写入报文发送到客户端。客户端选择第一个接到的 DHCP 报文，并把这一信息广播到网络中。

NAT （地址转换协议）

将局域网内部的私有地址转换为可以访问 Internet 的公网 IP。

一对多的 NAT 表现为一个运营商提供的公网IP，在网络出口的位置NAT将所有的内网网络设备转化为一个公网IP的形式，所有对公网的访问表现为一台主机。

OSPF （开放式最短路径优先）

内部网关协议用于在自治系统内决策路由。使用Dijkstra提出的最短路径算法。

OSPF通过路由器之间通告网络接口的状态来建立链路状态数据库，生成最短路径树，每个OSPF路由器使用这些最短路径构造路由表。路由器掌握了该区域上所有路由器的链路状态信息，也就等于了解了整个网络的拓扑状况。

RIP （路由信息协议）

RIP协议基于距离矢量算法（DistanceVectorAlgorithms），使用“跳数”(即metric)来衡量到达目标地址的路由距离。这种协议的路由器只关心自己周围的世界，只与自己相邻的路由器交换信息，范围限制在15跳之内，再远，它就不关心了。

BGP （边界网关协议）

BGP 边界网关协议（BGP）是运行于 TCP 上的一种自治系统（AS）的路由协议，是唯一能够妥善处理不相关路由域间的多路连接的协议。

当需要进行数据包的转发时，边界路由协议会帮助选择最佳的自治系统进行数据包的转发。比如，当同时接入了联通、电信、移动三大运营商的网络时，BGP会根据网络情况动态选择最佳的路线。
